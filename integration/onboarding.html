כניסה למערכת דיווח

מדיניות נוכחות — חשוב לדעת

אפשר לעבוד עד יומיים בשבוע מהבית, אבל לא יומיים רצופים (ראשון + חמישי נחשבים רצופים).
לכל צוות יש יום חובה במשרד: צוותי שולמית — יום שני, צוותי ימימה — יום רביעי.
כל חמישי מדווחים את התכנית לשבוע הבא במערכת דיווח המיקום.

ארכיטקטורה וסטנדרטים

ארכיטקטורת מערכות

עובדים עם שילוב של Microservices ו-Monolith, בהתאם למערכת. ממשקים נבנים כ-REST APIs או WCF בהתאם לצורך.

תשתיות

חלק מהמערכות רצות על AWS וחלק On-Prem (IIS, SQL Server). חשוב להכיר את שתי הסביבות ואת ההבדלים ביניהן.

הפרדת סביבות

קיימת הפרדה מוחלטת בין הסביבות. כל סביבה משרתת שלב אחר בתהליך:

DEV סביבת פיתוח — הרשאות מלאות למפתחים. כאן כותבים ובודקים קוד חדש.
TEST סביבת בדיקות — הרשאות מלאות לצוות QA, צפייה בלבד למפתחים.
PRE-PROD שלב אחרון לפני ייצור. משמשת גם לבדיקת באגים מייצור.
PROD סביבת ייצור — גישה מוגבלת. שינויים רק דרך תהליך העלאה מסודר.

קונבנציות ושמות

קיימים מסמכי סטנדרט לשמות — חובה לקרוא לפני כתיבת קוד או הקמת רכיבים בענן:

קונבנציות SQL Server כללי שמות לטבלאות, סכמות, אילוצים, SP ו-Functions
סביבה מאוחדת בענן AWS — סטנדרט שמות תבניות שמות לכל רכיבי הענן (S3, Lambda, ECS, IAM ועוד)

ניהול קונפיגורציה

הגדרות המערכות מנוהלות בצורה מרכזית. שינויי קונפיגורציה עוברים תהליך מסודר ומתועדים בכל סביבה בנפרד.

עבודה עם נתונים

SQL Server — כללי עבודה

כל העבודה מול בסיסי הנתונים מתבצעת לפי מסמך הקונבנציות. עיקרי הכללים:

הפרדה באמצעות סכמות (app, code, sec, cfg, rpt, audit) — לא משתמשים ב-dbo במערכות חדשות.
שמות טבלאות ועמודות ב-PascalCase. Primary Key תמיד <TableName>Id.
אין שימוש ב-Identity — רק SEQUENCE. אין קיצורים (Num/Descr) — מילים מלאות בלבד.
חובה להגדיר Foreign Keys בכל הטבלאות.

קיימות תבניות מוכנות למסמכי אפיון, מסמכי מערכת ותיעוד טכני. התבניות מצורפות למייל הקליטה — יש להשתמש בהן ככלי עבודה קבוע.

Best Practices לפיתוח

קוד קריא

קוד צריך להיות ברור גם למפתח שלא כתב אותו — שמות משתנים משמעותיים, מבנה לוגי מסודר, ללא קיצורים לא ברורים. קוד קריא מקל על תחזוקה ושיתוף ידע.

טיפול בשגיאות

יש לטפל בשגיאות בצורה מבוקרת — הודעות ברורות למשתמשים ולוגים מפורטים למפתחים. טיפול נכון משפר יציבות ומאפשר איתור תקלות מהיר.

ביצועים

קוד יעיל שמנצל נכון CPU, זיכרון וגישה ל-DB. יש להימנע מקריאות מיותרות ולבצע אופטימיזציה בנקודות קריטיות. ביצועים טובים משפיעים על חוויית המשתמש ועלויות.

Idempotency

פעולה אידמפוטנטית מחזירה את אותה תוצאה גם אם מבוצעת מספר פעמים. חשוב במיוחד בתהליכים מבוזרים או כשיש retry — מונע כפילויות ובעיות.

Retry נכון

מנגנון retry עבור כשלונות זמניים (קריאות רשת וכו'). חשוב לשלב backoff הדרגתי ולהגביל ניסיונות. retry לא נכון עלול לגרום לעומס או כפילויות.

תהליכי פיתוח (SDLC)

שלבי הפיתוח

התהליך מתחיל מאפיון דרישות, עובר לפיתוח, בדיקות ולבסוף העלאה לייצור. בכל שלב יש גורם אחראי ותוצרים נדרשים. אין קפיצות — מעבר בין שלבים רק עם אישור.

ניהול משימות — Azure DevOps

כל משימה מנוהלת ב-DevOps עם תיאור, קריטריוני קבלה והקצאה. חשוב לעדכן סטטוס באופן שוטף ולדווח על חסמים. זה מקור האמת לניהול העבודה.

Branching Strategy

כל פיתוח מתבצע ב-feature branch נפרד כדי למנוע התנגשויות. מיזוג ל-develop/main מתבצע רק אחרי בדיקות ואישור. שומרים על ענף ראשי יציב תמיד.

Code Review

כל קוד חייב לעבור בדיקה של מפתח נוסף לפני מיזוג. הבדיקה כוללת איכות קוד, עמידה בסטנדרטים וזיהוי בעיות. זה גם כלי מעולה לשיתוף ידע בצוות.

Definition of Done

משימה נחשבת "גמורה" רק כשכל הקריטריונים מתקיימים — פיתוח, בדיקות, תיעוד ואישור. ההגדרה מבטיחה אחידות ואיכות בכל המשימות.
